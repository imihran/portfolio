Foundational Data Layer Architecture (12 Metric Domains)

PySpark SQL Data Modeling Pipeline Orchestration Cloud Data Warehouse

Designed a comprehensive primitive data layer spanning 12 metric domains that serves as the single source of truth for all executive reporting, eliminating 11-20 hours/week of manual data preparation.

Problem

Executive reporting relied on manual data preparation across dozens of metrics with inconsistent definitions. Each report required custom joins and transformations, creating a maintenance burden and data quality risks. No centralized, governed data layer existed.

Approach

Designed a two-stage architecture: staging tables at item grain joined with a centralized attribute foundation table, then rolled up to category-level aggregations with time-frame comparisons. Led cross-functional design collaboration with 4 engineers to define pipeline architecture and data flow patterns.

Technical Details

  • 12 metric domains covering financials, supply chain, inventory, and customer metrics
  • Two-stage pattern: item-grain staging → category-level rollup with time-frame comparisons
  • Centralized attribute foundation table as single source of truth
  • Automated orchestration removing single-points-of-failure tied to individual contributors
  • Natural language query interface for self-service metric querying (in progress)
  • Migration from traditional data warehouse to modern lakehouse for scalability and governance

Impact

  • 9 of 12 metric domains validated and operational (75%)
  • Eliminated 11-20 hours/week of manual data preparation
  • Ensures data consistency across all vendor scorecards
  • Foundation table reused by multiple downstream projects
